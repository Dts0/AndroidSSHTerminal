## 🔴 Critical

### C1 — Host key 变更被静默接受，TOFU 形同虚设（MITM 漏洞）
- 位置：`app/src/main/java/com/sshtool/ssh/HostKeyVerifier.kt:56-66`
- 问题：`TrustingHostKeyRepository.check()` 在 `delegate.check()` 返回 `CHANGED`（已信任的 host 的 key 已变化）时原样返回 `CHANGED`，不触发确认对话框。服务器替换/DNS 劫持/中间人换 key 后，连接不会再次询问用户。
- 影响：首次信任之后攻击者换 key 即可绕过验证，击穿 host key pinning 的目的。

### C2 — trust-and-reconnect 存在 TOCTOU，pin 的 key ≠ 实际用的 key
- 位置：`app/src/main/java/com/sshtool/ssh/SSHConnection.kt:162-166`、`SSHConnectionManager.kt:110-123`
- 问题：`trustCurrentHostKey()` 读取第一次失败的 `session.hostKey`，但 `trustAndReconnect` 销毁 session 后重新握手，第二次握手的 key 可能不同。攻击者在两次握手间轮换 key，配合 C1 可完成 MITM。

### C3 — 旋转屏幕导致 SSH session 泄漏 + 终端白屏
- 位置：`app/src/main/AndroidManifest.xml:19-27`（无 `configChanges`）、`TerminalFragment.kt:603-609`、`SSHConnectionManager.kt:149-151`
- 问题：`MainActivity` 无 `configChanges`，旋转销毁重建 fragment。`onDestroyView` 只清本地 `sessionStates`，不碰单例 `SSHConnectionManager`，JSch session/channel/reader 协程继续运行，重建后开新连接。`reattachListener()` 是为恢复写的死代码，从未调用。
- 影响：每次旋转泄漏一个 SSH session；触发服务器 max-sessions 限制；用户看到空白终端 + 卡在 Connecting。

### C4 — Room v1、无迁移、无 fallback，改 schema 即让老用户启动崩溃
- 位置：`app/src/main/java/com/sshtool/data/repository/HostRepository.kt:30, 37`
- 问题：`@Database version = 1, exportSchema = false`，构建无 `addMigrations()`/`fallbackToDestructiveMigration()`。给 `Host` 加字段后 Room 在老用户启动即抛 `IllegalStateException`，丢失所有已保存 host。

### C5 — `EncryptedSharedPreferences` 无错误处理 + 用了 alpha 依赖
- 位置：`app/src/main/java/com/sshtool/data/crypto/PasswordStore.kt:16-26`、`app/build.gradle.kts:87`
- 问题：Keystore 失效（生物识别变更、OEM 恢复出厂、密钥轮换）时 `.create()` 抛异常，`HostRepository`/`SSHToolApp.onCreate` 无 try/catch → 启动崩溃 → 已存密码永久不可读。`security-crypto:1.1.0-alpha06` 是 alpha，API 不稳。

### C6 — `SSHConnection.send` 写 outputStream 无单写者约束，字节可能交错
- 位置：`app/src/main/java/com/sshtool/ssh/SSHConnection.kt:203-217`
- 问题：每次按键 `scope.launch` 新协程并发 `write`/`flush`，JSch ChannelShell OutputStream 非线程安全，字节流可交错损坏。
