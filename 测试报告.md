## 11. 已知限制与建议

1. **自动化字符输入限制**：`adb shell input text` 在 IME 激活状态下，部分字符会因 IME 拦截未到达 SSH。可使用 `input keyevent <ASCII>` 逐字符发送（更可靠但慢）。
2. **`screencap` 失效**：因 `FLAG_SECURE` 生效，需要用 `uiautomator dump` 读 UI 树，或通过 `PrintWindow` 截 Windows 窗口（绕过 secure flag）。
3. **真实键盘栏长按测试未做**：例如长按 Ctrl 切换"按住 Ctrl"模式（Termux 风格）。
4. **PTY resize 在软键盘场景下不生效**：见测试 7。
5. **删除 host 不清理 `trusted_host_keys.tsv`**：见测试 8 附加发现 — 重新创建同名 host 会复用旧指纹。

### 测试 6：服务宕机/keepalive 机制

- **测试方法**：在宿主机上对 dreamoftime0.com 的 `sshd` 进程发 SIGSTOP，挂起 30 秒，再 SIGCONT 恢复。
- **观测点**：`ServerAliveInterval=15` + `ServerAliveCountMax=3` 应在 45 秒无响应后判定掉线。
- **结果**：
  - 挂起 30s 期间终端内 `cat` 命令 hang 住但不断开（与预期一致，因为 30s < 45s 阈值）。
  - 持续挂起超过 45s 后，JSch 触发 disconnect 事件，UI 状态切换到红色圆点 + "未连接"。
  - 进程 SIGCONT 恢复后，**不会自动重连**（需要用户重新点连接）— 符合设计：避免"误恢复"产生状态混乱。
- **结论**：✅ keepalive 阈值正确，无误判，但缺乏自动重连（属合理权衡）。

### 测试 7：软键盘弹起时的 PTY resize

- **测试方法**：进入 terminal 屏幕，弹起软键盘，dump `termux_terminal_view` 的行列数（通过 `dumpsys SurfaceFlinger` 不易获取；改用 `uiautomator dump` 抓取布局）。
- **现象**：
  - 软键盘弹起后 IME 占据底部 ~600 px，terminal 视图的可见区域减少。
  - 但 Termux 的 `TerminalView` 依赖父布局的 `onSizeChanged`，本次测试中**未观察到自动 resize**（行数保持不变，文字被压扁）。
  - 用户必须手动触发"重新打开 terminal"或切走再切回才能看到正确行数。
- **结论**：⚠️ PTY resize 在软键盘场景下未生效。可能的原因：
  - `configChanges` 没有覆盖 `softInputMode` 变化（已在 manifest 中添加，但 `onConfigurationChanged` 未处理）。
  - 或 `TerminalView` 本身没监听 `onSizeChanged`（需进一步查看 Termux 源码）。
- **建议**：短期方案是在 `TerminalFragment.onResume` 显式 `terminalSession?.updateSize(rows, cols)`。

### 测试 8：host 编辑/删除生命周期

- **测试方法**：
  1. 长按 TestHost 卡片 → 进入 HostEditorFragment，验证字段预填。
  2. 点击 delete (btn_delete) → 验证"删除主机"确认对话框。
  3. 确认 → 验证 TestHost 从列表中消失 + DB row 被删除。
- **结果**：
  - ✅ **8a 编辑**：长按 TestHost 卡片后，editor 页面正确预填了 `TestHost / dreamoftime0.com / 22 / test / 密码`（截图 `edit-02-longpress.png`）。
  - ✅ **8b 删除确认对话框**：点击 btn_delete 后弹出"删除主机"标题 + 取消/删除按钮（截图 `del-01-after-tap-delete.png`）。
  - ✅ **8c 真删除**：点击"删除"按钮后，列表只剩 DemoHost。`sqlite3 ssh_tool_db` 查询确认 row 已删除：
    ```
    2|DemoHost|dreamoftime0.com|2222
    ```
- **附加发现**：
  - 删除 host 时**会**清理 `EncryptedSharedPreferences` 里的 password/private key/passphrase。
  - 但 `trusted_host_keys.tsv` **不会**同步清理 — 已删除的 host 的指纹仍残留（`h:508e50b85e7541b0ed5de688269fe56a4a45afce7b1cfef02ceb197ffafaa956` 对应被删的 TestHost:22）。这是 minor 设计缺陷 — 重新创建同名 host 后仍会复用旧指纹（"hash collision" 的副作用）。
  - 建议：删除 host 时同步调用 `hostKeyTrustStore.removeHost(host, port)`。
